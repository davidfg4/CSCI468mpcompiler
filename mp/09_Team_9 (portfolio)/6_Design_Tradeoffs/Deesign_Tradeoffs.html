Design Tradeoffs

Symbol Table Design

Our symbol table is implemented as a linked list of hashmaps where a symbol table at a given nesting level is represented as a hash map that uses a linked list to link to the next and previous symbol tables. This implementation is beneficial because of the constant-time performance provided by a hash map for insertion and deletion of records in any given symbol table. Since this is a linked list of hash maps the worst-case time it takes to look up a variable in the symbol tables is O(N) where N is the number of symbol tables. The fast operations and ease of implementation provided by these tools is the reason we chose them.

The only potential downside of this is that since this is a compiler we are writing, we cannot predict the initial capacity our hash map should have since our input could potentially be arbitrarily large. This means some rehashing may be necessary given a large program. However, for the purposes of this class the programs we are compiling are not large enough to matter, and performance constraints such as these have so far proven to be a non-issue.
